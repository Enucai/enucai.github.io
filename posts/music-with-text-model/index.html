How to Write Music Scores Using Text Models

2026-06-07enucai2 min read (320 words)
#music  #ai 

起因#

事情的起因是这样的：PKU 有一门神人课程名为“音乐与数学”，大作业的内容大概是将两段随机音乐发展为完整的曲子。

一直以为这个任务是容易的，直到开始做了才发现小组内没有人会写谱，也没有人有键盘或者吉他来录谱。随后就开始尝试使用各种 AI 工具（其实一开始就是指望着用 AI 的）。

问题来了：现在基本所有的音乐 AI 都是直接生成音频的，然后从音频里面扒谱是本质困难的工作。我尝试给定主旋律让 Suno AI 生成了一段音乐，但是随便一听就有至少十个声部，音轨分离之后尝试转成 MIDI 文件然后出现非常诡异不规则的谱子，旋律也全都不对了。

好这就是现在音乐 AI 的普适性问题：直接生成的音频听感很好，但是很难转回谱子（或者说这一项工作的难度很高）。

随后发现的是一个名为 AIVI 的东西，听说他可以直接生成谱子。但是这个东西的效果实在难评，丢给他主旋律生成的音乐和主旋律毫无关系，遂倒闭。

核心科技#

偶然间一个转折出现了：我在用 musescore 苦苦一个一个按音符，手写谱子的时候，突然发现有一种保存格式叫做 musicxml。保存了一下发现这个文件竟然是个 xml 格式的文本文件。也就是说，文本模型可以直接读取这个文件，同样也可以生成这个文件。

基于对强大的 gpt5.5 的自信，我打开了 codex 开始尝试，下面是我的第一个 prompt：

请阅读这个 musicxml 文件，这是一个主旋律的 demo，请以这一段旋律为主旋律，将其发展为一首完整的纯音乐。需要包含多个音部，配上和弦，可以对主旋律进行一定的扩写、变换，但是要在最后的乐曲中凸显出这一段主旋律。请直接写出可以聆听版本的 musicxml 文件。

然后 codex 老师真的生成了一个可以听的 musicxml 文件，而且还不赖。

后面的事就简单了，只需要在与 codex 的持续交互中优化听感就行。

codex 在写 bass、pad、鼓、和弦这些内容上还是非常擅长的，这也符合对大模型的直觉：毕竟这些内容都是机械的。

所以核心科技就是：有一种格式叫做 musicxml，因此文本模型也能写音乐。

技术总结#

下面是我让 codex 根据对话内容总结的一些内容，看着玩吧：

最近我做了一次挺有意思的实验：用文本模型辅助创作一首音乐。

这里说的“文本模型”不是专门的 AI 编曲软件，也不是那种输入一句 prompt 就自动吐出 mp3 的工具，而是像 Codex 这样的模型：它主要擅长理解文字、写代码、修改文件。但正因为乐谱本身也可以被表示成结构化文本，例如 MusicXML、MIDI 事件、脚本生成规则，所以它其实也可以参与音乐创作。

这篇文章想记录一下这次实验的过程：我是如何从一段随机生成的旋律中挑出一个主题，然后和文本模型一起，把它发展成一首比较完整的纯音乐小品的。

1. 音乐也可以是一种“文本”#

很多人第一次想到 AI 生成音乐，会想到直接生成音频。但这次我采用的是另一条路径：

旋律素材 -> MIDI / MusicXML -> 文本模型分析和修改 -> 生成乐谱 -> 播放试听 -> 继续反馈

也就是说，模型并不是直接“唱”出音乐，而是通过生成和修改乐谱文件来创作。

MusicXML 本质上是一种 XML 文件，里面记录了音符、时值、声部、乐器、和弦、力度等信息。MIDI 也可以被解析成音符事件。对文本模型来说，这些东西虽然不是自然语言，但仍然是结构化文本，只要配合脚本，就可以进行读取、分析和生成。

这就给了我们一个很有趣的创作方式：人负责听、判断、提出方向；模型负责快速实现谱面修改。

2. 从随机音乐中挑一个“种子”#

这次实验的原始材料是一段随机生成的粉色音乐。

随机生成的音乐往往有一个问题：它可能有一些局部听起来很有意思，但整体不一定像一首真正的曲子。因此第一步不是直接让模型扩写整首，而是先从随机结果里挑出一段有潜力的片段。

我和模型一起分析了原始旋律，最后选出了其中第 12 到第 16 小节，大约 5 小节的片段作为主题。

这段旋律有几个特点：

  • 节奏很跳，不是平滑歌唱型旋律；
  • 有不少突然的跳进；
  • 长度是 5 小节，有点不对称；
  • 虽然来自随机生成，但局部有可以记住的动机。

它不像传统古典旋律那样规整，但正因为不规整，反而有一种独特的随机感和机敏感。

于是问题变成了：这样的旋律适合发展成什么风格？

3. 不要强行古典化#

一开始我也尝试让模型做几种不同风格的 demo，比如钢琴小品、爵士/funk、电子 groove 等。

试听之后很明显：这个旋律不太适合写成太传统、太端庄的古典小品。因为它本身的魅力就在于节奏跳跃、句法不规则。如果强行配上平稳的古典伴奏，反而会显得别扭。

后来方向逐渐明确：它更适合现代电子律动或者游戏配乐风格。

这种风格有几个好处：

  • 稳定的鼓点可以托住跳跃旋律；
  • bass groove 可以提供方向感；
  • synth lead 可以让主题显得明亮；
  • 主题的不规则性不会被视为缺点，反而会变成个性。

这也是使用文本模型创作时很重要的一点：不要只问“你能不能帮我扩写”，而要不断判断“这个材料本身适合什么”。

4. 让模型生成乐谱，而不是只给建议#

确定方向后，我让模型开始生成 MusicXML。

一开始的版本通常不会特别好。模型可以很快写出一个完整结构，比如：

Intro - A - Transition - B - Breakdown - Development - Climax - Coda

也能生成 bass、pad、鼓、和弦、主旋律等多个声部。但问题是，第一次生成的东西往往“结构上是完整的，听感上是平的”。

这也是文本模型创作音乐的典型状态：它很擅长搭框架，但不一定第一次就能判断哪里拖沓、哪里单调、哪里没有高潮。

所以真正有效的工作方式不是“一次生成最终作品”，而是：

生成一版 -> 听 -> 指出问题 -> 修改 -> 再听 -> 再修改

这有点像和一个不会累的编曲助手合作。

5. 迭代比 prompt 更重要#

这次创作真正变好的地方，基本都来自迭代。

比如早期版本有一个 B 段和 transition，结构上看起来合理，但听起来很长，而且缺少变化。我指出之后，模型把整体缩短，删掉冗余段落，把 development 提前，变成更紧凑的结构。

后来又发现高潮不够突出。模型最初的做法只是把主题升高八度、加厚声部，但听起来并没有真正“起来”。于是我要求它重新处理 development 到 climax 的连接，不要在高潮前放一个太长的 breakdown，而是改成 pre-climax bridge，只短暂抽气，然后直接进入高潮。

再后来发现主旋律每次出现都太像，听起来像复制粘贴。我又要求模型做轻微变奏：A 段保持原主题，development 加入 G 调色彩，climax 改成高音区变奏，而不是简单高八度重复。

这些修改都不是靠一个神奇 prompt 完成的，而是靠反复听、反复指出问题。

6. 人类负责审美，模型负责实现#

这次实验给我最大的感受是：文本模型很适合做“实现者”，但不应该完全交给它做“审美裁判”。

比如它很擅长：

  • 写 MusicXML；
  • 生成多声部乐谱；
  • 根据要求改和弦；
  • 调整 bass pattern；
  • 修改鼓点；
  • 批量生成不同版本；
  • 保持文件结构正确。

但它不总是擅长：

  • 判断一段是不是太长；
  • 判断高潮是否真的有情绪推进；
  • 判断主旋律是否被伴奏盖住；
  • 判断某个段落是否听起来像流水账；
  • 判断整体风格是否统一。

这些更多需要人来听。

在这次创作里，我的角色更像是导演：我不一定能手写出每个声部，但我可以判断“这里太碎”“这里太空”“这里主旋律不清楚”“这里鼓点不够”“这里和声太单一”。模型则负责把这些判断落实到乐谱中。

7. 为什么 MusicXML 很适合这种合作#

如果只是让模型直接生成音频，修改会比较困难。你可能只能说“这里再激烈一点”“这里温柔一点”，但具体发生了什么不容易控制。

MusicXML 的好处是它非常可编辑：

  • 想改某个声部，可以直接改对应 part；
  • 想调整某小节，可以定位到 measure；
  • 想改变和弦，可以改 harmony；
  • 想调整主旋律，可以改 note；
  • 想加动态，可以写入 mf、f；
  • 想加表情，可以写入 accent、tenuto、staccato。

也就是说，音乐被拆成了可操作的结构。

这对文本模型非常友好，因为它本来就擅长处理结构化文本和代码。很多时候，我并不是让它“凭空创作音乐”，而是让它写脚本生成乐谱。这样既能快速迭代，又能保留可控性。

8. 最终作品的大致结构#

最后形成的版本大约 40 小节，结构是：

Intro pulse
引入性片段
A - varied theme
Transition / gentle lift
Development - G color
Pre-climax bridge
Climax - varied theme in G
Coda - return to C

编制大致包括：

  • 主旋律 synth lead；
  • 轻微加厚主旋律的 support；
  • mallet ostinato；
  • chord hits；
  • warm pad；
  • expressive bass；
  • drum kit。

整体风格可以概括为：现代电子律动风格的游戏配乐小品。

它保留了原随机旋律的跳跃感，又通过鼓点、低音、和声和结构把它组织成比较完整的作品。

9. 这种方法适合什么人？#

我觉得这种方法特别适合几类人：

  1. 有旋律想法，但不会完整编曲的人；
  2. 会判断听感，但不熟悉 MusicXML 或配器的人；
  3. 想把随机生成材料发展成完整作品的人；
  4. 想快速试很多版本的人；
  5. 会一点代码，愿意把音乐当成结构化数据处理的人。

它不一定能直接生成专业级作品，但非常适合做草稿、实验和快速迭代。

10. 一些经验总结#

这次实验之后，我觉得比较有效的做法是：

  • 不要一上来就让模型写完整曲子，先选一个主题；
  • 先判断主题适合什么风格；
  • 让模型生成多个方向的 demo；
  • 每次反馈尽量具体，比如“高潮不够突出”“bass 太机械”“主旋律被盖住”；
  • 不要怕多版本迭代；
  • 最后一定要靠人耳判断，而不是只看谱面结构。

文本模型生成音乐乐谱，本质上不是“一键作曲”，而是一种新的协作方式。

它把很多技术门槛降下来了：写谱、扩展声部、生成 MusicXML、调整结构，都可以很快完成。但音乐最终是否成立，仍然取决于人的听觉判断和审美取舍。

这大概也是目前 AI 创作最有意思的地方：它不是替你完成所有事情，而是让你能更快地尝试、失败、修改，然后逐渐靠近你想要的东西。
